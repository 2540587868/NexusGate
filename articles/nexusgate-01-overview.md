# NexusGate：用 Go 打造轻量级高性能 API 网关
> 「用 Go 构建网关」专栏第 1 篇。本文介绍 NexusGate 的项目定位、核心特性、技术选型与设计哲学。

---

## 为什么需要又一个 API 网关？

当你审视现有的 API 网关生态，会发现一个有趣的断层：

| 网关 | 特点 | 问题 |
|------|------|------|
| Nginx | 极致性能，C 语言 | 配置是 DSL，扩展性差，动态路由需 Lua |
| Kong | 插件丰富，基于 Nginx | 依赖 PostgreSQL/Cassandra，部署重 |
| Envoy | xDS 动态配置，C++ | 架构复杂，学习曲线陡峭 |
| Traefik | Go 实现，自动发现 | 配置模型复杂，性能不如 Nginx |
| 自研网关 | 完全定制 | 从零开始，工程量大 |

**NexusGate** 的核心主张：**用最少的依赖，构建最清晰的网关内核。**

不是要替代 Nginx 或 Envoy，而是提供一个**可理解、可定制、可嵌入**的 Go 原生网关内核——你可以在 30 分钟内读完所有源码，理解每一个设计决策。

## 项目定位

NexusGate = **Nexus**（连接点）+ **Gate**（门控）

它是一个 **轻量级 API 网关内核**，专注于回答以下问题：
- 如何将请求高效地路由到正确的后端？
- 如何在路由前后插入限流、熔断、CORS 等治理逻辑？
- 如何在不重启的情况下更新路由和后端配置？
- 如何在后端故障时自动摘除并恢复？
- 如何观测网关自身的运行状态？

**不是什么**：NexusGate 不做服务发现（不依赖 Consul/Etcd）、不做协议转换（不支持 gRPC-HTTP）、不做分布式追踪（不集成 Jaeger/Zipkin）。这些是上层系统的职责，NexusGate 只提供内核原语。

## 核心特性

### 🔀 分片引擎

| 维度 | 设计 |
|------|------|
| 分片策略 | 基于 TenantID 的 FNV-1a 哈希取模 |
| 分片数量 | 8 个固定分片（编译期确定） |
| 队列模型 | 每分片独立带缓冲 channel |
| 背压机制 | 队列利用率 >90% 时启动 slowRecover |
| 隔离性 | 同租户请求串行化，不同租户完全并行 |

8 个分片意味着：不同租户的请求互不干扰，一个租户的流量尖峰不会阻塞其他租户。这是**租户级隔离**的最低成本实现。

### 🧭 四策略路由

| 策略 | 算法 | 适用场景 |
|------|------|----------|
| 一致性哈希 | 150 虚拟节点 + CRC32 | 有状态服务，会话保持 |
| 加权轮询 | Nginx 平滑 SWRR | 无状态服务，按权重分配 |
| 最少连接 | 原子计数器实时追踪 | 长连接服务，负载均衡 |
| Header 路由 | 请求头匹配后端 Meta | 金丝雀发布，版本路由 |

四种策略覆盖了网关路由的绝大多数场景，每种都有明确的最佳实践。

### 🔗 链式中间件

```
Trace → AccessLog → RateLimit → CORS → CircuitBreaker → Handler
```

- **不可变链**：`Use()` 返回新 Chain，不修改原链
- **反向包装**：注册顺序即执行顺序
- **零分配**：中间件函数复用，无每次请求的堆分配

### 🔄 配置热更新

- **双缓冲 Store**：dirty/clean 分离，读者无锁访问已提交配置
- **文件轮询**：5 秒检测配置文件 ModTime 变化
- **原子切换**：Router 通过 `SwapRoutes` 一次性替换路由表
- **回调保护**：每个回调独立执行，带 `recover()` 防止 panic 传播

### 🏥 健康检查

- 定期 HTTP GET 后端 `/health` 端点
- 连续失败达阈值自动摘除
- 恢复后自动加回
- 与 Router 和 Proxy 联动更新

### 📊 可观测性

- **零依赖 Prometheus 指标**：自研直方图、计数器、摘要
- **请求级追踪**：自动生成 TraceID，贯穿全链路日志
- **结构化访问日志**：方法、路径、状态码、耗时（微秒级）
- **后端维度指标**：每个后端的请求数、失败数、延迟统计

## 技术选型

| 类别 | 选择 | 版本 | 理由 |
|------|------|------|------|
| 语言 | Go | 1.22+ | 原生并发、交叉编译、标准库强大 |
| 配置解析 | gopkg.in/yaml.v3 | v3 | 唯一外部依赖，YAML 人类可读 |
| HTTP 客户端 | net/http | 标准库 | Transport 连接池，Keep-Alive 复用 |
| 哈希算法 | hash/fnv | 标准库 | FNV-1a 分布均匀，计算极快 |
| 一致性哈希 | hash/crc32 | 标准库 | CRC32 IEEE 表，虚拟节点映射 |
| 日志 | log/slog | 标准库 | 结构化日志，Go 1.21+ 内置 |
| 指标 | 自研 | - | 零依赖，atomic + sync.Map |
| 前端 | 无 | - | 纯后端网关，指标通过 /metrics 暴露 |

**外部依赖：仅 1 个**（`gopkg.in/yaml.v3`）。这是 NexusGate 的设计红线——每一个新依赖都需要充分论证其必要性。

## 设计哲学

### 1. 内核极简，扩展在外

NexusGate 的内核只做三件事：**接收请求、路由分发、代理转发**。限流、熔断、CORS 等都是中间件，可以自由组合或替换。配置热更新是 Store + Watcher 的组合，不是内核的一部分。

### 2. 显式优于隐式

- 错误码是枚举常量，不是魔法数字
- 中间件链是显式组合，不是注解扫描
- 路由策略是配置声明，不是代码逻辑
- 健康检查是主动探测，不是被动猜测

### 3. 无锁优先

- 分片引擎：每个 Shard 独立队列，无全局锁
- 路由查询：`sync.RWMutex` 读多写少
- 指标收集：`atomic.Int64` + `sync.Map`
- 配置读取：`atomic.Value` 无锁访问

### 4. 渐进式复杂度

- 单后端：最简配置，开箱即用
- 多后端：选择路由策略，配置权重
- 多租户：启用限流，设置 TenantID
- 生产级：开启熔断、健康检查、指标暴露

每一层复杂度都是可选的，不会因为不需要的功能而增加认知负担。

## 代码规模

| 模块 | 文件数 | 代码行 | 说明 |
|------|--------|--------|------|
| gateway | 2 | ~230 | 分片引擎 + 类型定义 |
| router | 6 | ~450 | 四种路由策略 |
| proxy | 3 | ~280 | 代理转发 + 重试 + 后端追踪 |
| middleware | 7 | ~650 | 六种中间件 + 链式组合 |
| config | 2 | ~400 | 配置存储 + 文件监控 |
| lifecycle | 3 | ~300 | 优雅关闭 + 健康检查 + 可恢复协程 |
| httparser | 1 | ~470 | HTTP 请求解析 + 响应写入 |
| cmd | 1 | ~235 | 入口 + 连接处理 |
| **总计** | **25** | **~3015** | **不含测试和配置文件** |

3000 行代码，一个下午可以通读全部源码。这是 NexusGate 最大的优势——**可理解性**。

## 快速开始

```bash
# 构建
go build -o nexusgate ./cmd/nexusgate

# 启动（默认配置）
./nexusgate -config configs/nexusgate.yaml

# 带指标端点
# 在配置文件中设置 server.metrics_listen: ":9090"
# 访问 http://localhost:9090/metrics 获取 Prometheus 指标
```

最小配置示例：

```yaml
server:
  listen: ":8080"

routes:
  - match:
      path_prefix: "/api"
    backends:
      - address: "127.0.0.1:3000"
        weight: 1
    strategy: weighted_round_robin
```
